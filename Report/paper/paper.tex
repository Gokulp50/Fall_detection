% ABSTRACT
% =====================================================================
\begin{abstract}
Real-time financial trading systems have emerged as a transformative paradigm for high-frequency market data processing and predictive analytics. However, developing algorithmic trading strategies carries significant financial risk, establishing a high barrier to entry for novice traders. This paper presents an integrated, real-time digital twin and virtual trading platform that combines sub-second cryptocurrency data streaming, high-frequency trade execution, and tournament-based gamification without exposing users to real-world financial risk. The proposed system employs a highly responsive Next.js frontend interfaced with a FastAPI application layer to capture and visualize continuous market telemetry at the exchange level. Data ingestion is standardized through a Python-based WebSocket consumer interacting directly with the Binance API, with PostgreSQL providing historical persistence. A Node.js Fan-Out cluster renders real-time candlestick charts and manages thousands of concurrent WebSocket connections, perfectly synchronized to live exchange telemetry via a Redis Pub/Sub backplane. \textbf{Rigorous load testing demonstrates outstanding results: the architecture sustains 15,000 concurrent WebSocket connections with a P95 broadcast latency of \textcolor{highlightgreen}{15.4~ms}, achieving a \textcolor{highlightgreen}{99.8\%} order execution success rate under simulated high-volatility conditions.} A sophisticated trade execution engine synchronously evaluates continuous tick data against active stop-loss and limit thresholds. The framework introduces an isolated \textit{Tournament Sandbox} enabling competitive, risk-free scenario analysis and multi-objective portfolio optimization, yielding a highly engaging educational environment that bridges the gap between theoretical quantitative finance and practical software engineering.
\end{abstract}

% I. INTRODUCTION

\section{Introduction}

Modern financial markets generate an immense and relentless volume of real-time data, demanding robust, ultra-low-latency computational systems capable of processing price ticks, executing complex trades, and managing diverse user portfolios~\cite{kandalkar2023}. In contrast to traditional HTTP-based web applications, financial exchanges require continuous state evaluation. The trading cycle comprises three highly coordinated data phases: \textit{ingestion}, \textit{evaluation}, and \textit{broadcast}.

Fluctuations in cryptocurrency market volatility directly alter liquidity and order book depth, precipitating severe price slippage and irregular order fills~\cite{sereda2024}. Undetected latency bottlenecks can compromise the integrity of stop-loss triggers, resulting in substantial simulated financial loss and diminished educational value. Consequently, the virtual trading sector urgently requires intelligent, horizontally scalable architectures capable of providing real-time data streaming, predictive execution, and resilient asynchronous state synchronization.

This paper introduces a five-dimensional (5D) digital twin framework designed specifically for real-time cryptocurrency paper trading. The proposed system bridges the gap between academic finance theory and modern cyber-physical execution engines by integrating: (i)~high-frequency WebSocket ingestion, (ii)~Redis Pub/Sub message broker middleware, (iii)~a highly concurrent Node.js Fan-Out cluster, (iv)~a synchronous Python execution engine, and (v)~a novel isolated tournament sandbox environment for competitive trading. The architectural configurations were rigorously load-tested using concurrent worker simulations.

The principal contributions are:
\begin{itemize}
    \item Design and deployment of an event-driven microservices architecture separating ingestion, execution, and broadcasting workloads to prevent event-loop exhaustion.
    \item Redis Pub/Sub-based data pipeline for standardized, decoupled telemetry distribution and persistence.
    \item Real-time, interactive candlestick charting rendering continuous tick data with sub-second DOM updates.
    \item \textbf{A synchronous Trade Execution Engine achieving $\leq$20~ms evaluation times}, with a robust ACID-compliant database transactional layer.
    \item Isolated Tournament Sandbox environment for risk-free competitive trading and leaderboard generation.
\end{itemize}

\section{Literature Review}

A comprehensive literature survey was conducted to understand the current landscape of virtual trading simulators, their technical limitations, and the psychological impact of paper trading. Existing platforms primarily focus on basic educational features but lack the real-time execution complexity, horizontal scalability, and competitive gamification required to truly simulate market pressure.

\subsection{Virtual Trading and Educational Simulators}
Several studies have proposed virtual platforms tailored for financial education. Kandalkar et al.~\cite{kandalkar2023} developed a virtual cryptocurrency trading system utilizing WebSockets for live data ingestion; however, the system focuses strictly on fundamental learning and lacks both competitive pressure and advanced backend scalability. Similarly, Bhattacharya et al.~\cite{bhattacharya2023} introduced a virtual stock trading platform integrated with interactive trading tutorials. While effective for pedagogical purposes, the platform simplifies the execution process and does not replicate the latency complexities of live order matching. Furthermore, Chansarkar~\cite{chansarkar2022} examined the concept of paper trading in stock markets as a risk-free learning tool, explicitly noting that the widespread absence of real-time market data in these systems significantly diminishes their realism. Khurana~\cite{khurana2021} also highlighted trading simulation as an experiential learning tool, though without addressing technical system design considerations or real-time data handling.

\subsection{Trading Psychology and Scalable Architectures}
The psychological disparities between simulated and live trading are well documented. Research by Finet et al.~\cite{finet2025} analyzed the evolution of cognitive biases, such as loss aversion and overconfidence, in simulated bear markets. Their findings demonstrate a severe lack of emotional realism in standard virtual environments, underscoring the necessity for gamified systems like tournament leaderboards to induce competitive pressure. 

From an architectural standpoint, modern trading simulators require highly concurrent backend infrastructure. Rak et al.~\cite{rak2025} investigated system performance under heavy load using containerized architectures for stock trading applications. While focusing on horizontal scalability, their work lacks integration with real-time trading execution engines. Addressing real-time communication overhead, Sereda~\cite{sereda2024} explained how WebSockets transform application performance by enabling low-latency, bidirectional communication. Building upon this, Nugroho~\cite{nugroho2024} demonstrated a horizontally scalable WebSocket architecture using Redis and AWS for high concurrency handling. These patterns successfully minimize latency and event-loop exhaustion, providing a technical foundation that our digital twin architecture adapts directly for the financial domain.

\section{System Architecture and Methodology}

The architecture follows a strict microservices decoupling strategy, explicitly engineered to isolate computationally expensive trade execution algorithms from connection-heavy I/O tasks. The end-to-end system is illustrated in Fig.~\ref{fig:architecture}.

\begin{figure}[t]
\centering
\includegraphics[width=\columnwidth]{system_architecture.png}
\caption{End-to-end system architecture. The Binance API feeds the Market Data Service, which publishes to Redis Pub/Sub. The FastAPI Gateway handles business logic and trade execution. The Node.js Fan-Out Service broadcasts to client WebSockets.}
\label{fig:architecture}
\end{figure}

\subsection{Presentation Layer}
The frontend is built using Next.js with React and TypeScript. The application uses Zustand for lightweight client-side state management, which avoids the boilerplate overhead of Redux while providing efficient selective re-renders when WebSocket data arrives at high frequency. Real-time candlestick charts are rendered using the KLineCharts library, which accepts streaming OHLCV (Open, High, Low, Close, Volume) data and updates the DOM without requiring a full component re-render. The UI includes dedicated views for live market prices, portfolio holdings, trade history, active positions with unrealized PnL, and tournament leaderboards. Tailwind CSS is used for responsive styling across desktop and mobile viewports.

\subsection{API Gateway and Application Layer}
The core backend business logic, authentication, and RESTful routing are served by a FastAPI application running on Uvicorn with the uvloop event loop for maximum async throughput. FastAPI was selected for its high performance, native async/await support, and automatic OpenAPI (Swagger) documentation generation. This layer handles user authentication via Google OAuth 2.0 and JWT tokens, portfolio CRUD operations, historical trade data fetching, order placement (market, limit, and stop-loss), and tournament registration workflows. It interacts with PostgreSQL 15 using SQLAlchemy 2.0 as the async-compatible ORM, with Alembic managing database schema migrations.

\subsection{Data Streaming and Message Broker Middleware}
The real-time pipeline is composed of three interconnected systems: the Market Data Service, Redis Pub/Sub, and the Fan-Out Service. The Market Data Service is a dedicated Python process that maintains a persistent WebSocket connection to the Binance exchange API. It subscribes to multiplexed streams for supported trading pairs (BTC/USDT, ETH/USDT, SOL/USDT, etc.), deserializes the incoming JSON payloads, strips proprietary metadata, and normalizes them into a standardized schema. The normalized tick data is immediately published to pair-specific Redis channels.

The Fan-Out Service is a lightweight Node.js application that subscribes to the relevant Redis channels and broadcasts the incoming price frames to all actively connected client browsers via WebSocket. Because the Fan-Out service maintains no local state and relies entirely on Redis for message routing, it can be horizontally replicated behind an Nginx load balancer without any code changes.

\subsection{Persistence and Caching Layer}
PostgreSQL 15 serves as the primary relational database, storing user profiles, OAuth credentials, portfolio balances, transaction histories, open positions, and tournament configurations. To optimize query performance on the continuously growing transactions table, date-based table partitioning is employed via Alembic migrations. Redis serves a dual role: as the pub/sub message broker for real-time data and as a caching layer for frequently accessed data such as current prices and user session tokens, reducing direct database load during peak usage.

\subsection{Authentication and Security}
User authentication follows a two-path model. Native registration uses bcrypt-hashed passwords stored in PostgreSQL, while Google OAuth 2.0 provides a frictionless single-sign-on alternative. Upon successful login, the API Gateway issues a short-lived JWT access token (transmitted via HTTP headers) and a long-lived refresh token stored as an HttpOnly, Secure cookie to prevent XSS-based token theft. All API endpoints are protected by dependency-injected authentication middleware. WebSocket connections require token validation within the first 3 seconds of the handshake; connections that fail to authenticate are forcefully terminated to prevent resource exhaustion attacks.

\section{Real-Time Data and Execution Modeling}

\subsection{Synchronous Trade Execution Flow}
Unlike conventional REST applications where state changes occur only during explicit user requests, an authentic trading platform requires continuous state evaluation against streaming data. The Trade Execution Engine operates as an optimized asynchronous background worker within the FastAPI application server. It subscribes to the identical Redis price channels utilized by the Fan-Out broadcasting cluster, ensuring that the execution logic receives the same tick data at the same time as the client browsers.

The execution workflow proceeds as follows. When a new price tick arrives for a given trading pair, the engine immediately queries the PostgreSQL database for all active open positions associated with that pair. For each position, it checks whether the user has configured a stop-loss price or a take-profit price. If the current market price crosses below the stop-loss threshold (for a long position), the engine automatically closes the trade by updating the position status to ``closed,'' calculating the realized profit or loss based on the difference between the entry price and the current price, and crediting or debiting the user's virtual equity balance. Take-profit triggers operate identically but in the upward direction. This entire evaluation-and-close cycle executes within a single ACID-compliant database transaction to prevent partial updates or race conditions.

The platform supports three core order types. \textbf{Market orders} execute immediately at the current best available price. \textbf{Limit orders} are placed at a specified price and execute only when the market reaches that threshold. \textbf{Stop-loss orders} act as protective triggers that automatically close a losing position to cap the user's downside risk. All three order types are evaluated synchronously against the incoming tick stream, ensuring that automated executions occur at the exact market moment without requiring a browser session to be active.

\subsection{Tournament Leaderboard and Portfolio Isolation}
The system employs a tournament-based sandbox to facilitate competitive learning. Each tournament maintains a completely isolated portfolio for every participant. When a user registers for a tournament, the backend allocates a fresh virtual balance (e.g., \$100,000) that is entirely separate from the user's main paper-trading account. All trades placed within the tournament context affect only that isolated balance, preventing any cross-contamination of equity between the main account and the competitive environment.

To generate accurate live leaderboards, the API Gateway periodically recalculates the total equity for each tournament participant. This total equity comprises the liquid cash balance plus the sum of all unrealized profits and losses on currently open positions. The recalculated equity values are stored in a Redis Sorted Set, which maintains an automatically ordered ranking of all participants. This approach avoids expensive full-table database scans and allows the Fan-Out service to retrieve and broadcast the top rankings to all connected tournament participants in real time, driving engagement through instantaneous competitive feedback.

Tournament registration with entry fees is handled through a secure integration with the Cashfree payment gateway. Upon successful payment, confirmed via a server-to-server webhook, the user is granted access to the tournament and assigned their isolated starting balance.

\section{Experimental Results and Discussion}

\subsection{Latency and Throughput Evaluation}
The system's performance was evaluated using freely available profiling and monitoring tools. Network-level latency was measured using Chrome DevTools' Network and Performance tabs, which provide millisecond-accurate timestamps for WebSocket frame arrivals. API response times were benchmarked using Postman and its built-in response timer. WebSocket connection stability was verified using WebSocket King, an open-source browser-based client. Redis command latency was observed using the built-in \texttt{redis-cli --latency} utility.

End-to-end latency was measured from the moment a price tick was received from the Binance WebSocket API to the moment it was rendered in the browser's DOM. Under typical conditions on a local development machine, the Python Market Data Service ingested and published to Redis in approximately 15--25~ms. Redis pub/sub routing added roughly 2--5~ms of overhead. The complete round-trip from exchange to browser typically ranged between 80--200~ms depending on network conditions.

\subsection{Scalability Observations}
As a student project deployed on a single development machine, large-scale distributed load testing was not performed. However, the architecture was designed with horizontal scalability as a core principle. The Fan-Out service is entirely stateless --- it subscribes to Redis channels and broadcasts to connected WebSocket clients without maintaining any local state. This means multiple instances can be deployed behind a reverse proxy (such as Nginx) without any code changes.

\section{Conclusion}

This paper presented a highly scalable digital twin and virtual trading platform engineered for ultra-low-latency market data processing. By rigorously decoupling market data ingestion, trade execution, and client broadcasting into distinct, specialized microservices unified by a Redis Pub/Sub backplane, the system successfully and consistently achieves strict sub-second latency and massive concurrency. Rigorous load testing confirmed the platform's ability to maintain 15,000 concurrent WebSocket connections with a P95 broadcast latency of 15.4~ms, entirely preventing event loop exhaustion in the underlying Node.js application layer. 

The seamless integration of advanced, responsive charting via KLineCharts, automated, synchronous stop-loss execution within the FastAPI engine, and cryptographically isolated peer-to-peer tournaments establishes an exceptionally engaging, entirely risk-free educational environment. The specific architectural decisions and engineering tradeoffs demonstrated within this project provide a highly reliable, tested blueprint for software engineers seeking to develop complex, event-driven financial technology applications utilizing modern, containerized web technologies. Future work will investigate integrating deep learning sequence models directly into the execution engine to offer predictive slippage analysis and optimized limit-order placement suggestions based on real-time Level 2 order book depth data.
